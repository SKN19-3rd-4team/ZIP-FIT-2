0. 최종 검토 수정 사항

1. 데이터 확인 및 수정 (공고날짜,D-day,기관에 따른 공고 설정)
1.1. mockData 정밀하게 수정.
1.2. 수정사항, 각 공고에 해당하는 지역 설정 올바르게 할 것.
- LH : 전체 지역 공고
- SH : 서울 지역내 공고만 (공고명에서 서울 지역 내 공고는 SH에 포함한다)
- GH : 경기 지역내 공고만 (공고명에서 경기 지역 내 공고는 GH에 포함한다)
예) 현재 잘못된 예시
```
세종 전세임대주택 입주자 모집공고
모집 기간 
2025.11.16 ~ 2025.12.10 마감 ->SH 태그
```
-> 세종 -> 은 '경기 지역' 이므로 -> GH 태그, 혹은 LH 태그

1.3. 모집 기간 수정
- 모집기간을 2024년-현재, 2025년 12월 11(오늘 날짜로 기준) 다양하게 mockData 설정하기
1.4. D-day 설정
- 올바르게 수정된 모집 기간 데이터를 바탕으로 현재 날짜를 항상 불러와서 D-day 올바르게 계산하기
- 마감 된 공고(현재 날짜에 지나간 공고 데이터)는 이전과 동일하게 '마감' 으로 표기할 것.
1.5. 상태 태그 설정
- 현재 mockData 또한 현재 날짜를 불러와서 적용할 것.
예를 들어 현재 mockData는 아래와 같은 오류가 있음.
```
세종 전세임대주택 입주자 모집공고
모집 기간 
2025.11.16 ~ 2025.12.10 마감 ->공고예정 태그
```
-> today 불러왔을때, 마감 ok
그러나, '공고예정' 태그는 아님, '마감' 태그가 달려야함.

2.1. list 페이지의 '공고 카드' 정렬 규칙
기본적인 정렬 규칙은 우리가 논의해서 정한 '공고중, 접수중, 마감, 공고예정'
list 첫 페이지 -> 3개의 필터 적용 안됨.
정렬 규칙 적용됨.
2.2. 필터 변경해서 다시 sorting 해서 카드를 보여줄 시 -> 정렬 규칙 적용 안됨.
예) 기관 필터 선택시 사용자 'LH 한국토지공사' 선택
-> 공고 카드 filtering 되어 화면에 노출 -> 정렬 규칙 따르지 않음.
-> 마찬가지로 공고 유형 필터만 다른 옵션을 선택했을 시도 정렬 규칙 따르지 않음.
-> 마찬가지로 상태 필터만 옵션 선택했을때도 정렬 규칙 따르지 않음.

3. 공고 카드 url -> 우리 설계상 실제로는 API에서 호출한 공고url 로 접근
mock 데이터는 현재 실제 공고가 아니므로 url 이 없다.
각 기관 태그에 따라서 공고에 대해서 url을 아래와 같이 mockData url 링크를 접속한다.
공고에 맞는 각 기관 사이트 링크 (아래 3개, LH, GH, SH)
https://apply.lh.or.kr/lhapply/main.do
https://www.gh.or.kr/
https://www.i-sh.co.kr/main/index.do

4. 히스토리 채팅 클릭시 -> 로딩 화면만 나오고 다음 화면 출력되지 않음.
4.1. 다음 화면 : 실제 채팅한 데이터 내역
4.2. mockData로 가상의 대화 시나리오 간단히 작성해줄 것.
-> 시나리오는 실제로 호출되는 API 값에 맞춰서 작성
-> 실제로는 API에서 호출한 값을 호출해야함.
-> 시나리오 예시, 이전 우리 설계에 맞춰서 진행
- 공고 요약 출력 예시
- 공고 비교 출력 예시
- 기타 질문 출력 예시
- ... 등
-> 나중에 실제 API를 통해서 응답받은 데이터일 경우, 우리가 어떻게 출력하겠다라는 것을 규정해야함.
-> 이전에 설계 단계에서 언급함.

5. 최종적으로 mockData는 실제 우리가 API에서 호출하는 response 값에 맞춰서 데이터를 가공해야한다.
다만, 우리 api를 통해서 전달 받는 규격과 응답 형태가 정보가 부족할 수도 있다.
-> 만약 프론트에서 필요로 한다면, 백엔드에 요청해서 response 해주는 값들을 일부 수정할 수 있음.
그래서 한 번에 나중에 API 데이터로 갈아끼울 수 있어야한다. 최종 검토를 진행할 것.
